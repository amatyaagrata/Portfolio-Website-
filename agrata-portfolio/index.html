How I work

Quality is a habit, not a checklist.

An engineering style shaped by testing, data, and building things people can rely on.

Evidence before opinions

Research the data, understand the edge cases, then write the code. I prefer decisions I can defend with metrics.

Test early, test often

Whether it's unit tests, Postman API checks, or manual QA — I validate at every stage, not just before deployment.

Clean code is kind code

I write code that the next developer can read. OOP principles, meaningful names, and documentation are non-negotiable.

Ship the working path

I choose the simplest architecture that solves the real problem, then iterate. No over-engineering, no gold-plating.
